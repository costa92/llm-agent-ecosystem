## 3. 流程评估

### 3.1 上一份 retro 建议的兑现情况

| 上份 §7 建议 | 兑现状态 |
|---|---|
| 直接接 v1.6 候选讨论 | ✅ 没重做分析阶段，直接进 discuss |
| otel-side OnGenerateUsage 消费 优先 rag 新 minor | ✅ otel v0.3.0 第一时间出，rag v1.6 后接 |

| 下游同步自动化 `cmd/bump-downstream` | ❌ 仍手工（execute agent 流程已稳定，工具 ROI 低于预期） |

兑现率 3/4。第 4 项的不兑现是合理的——execute agent + 同步 polling loop 已稳定到「单次 ~3-5 分钟」，再加一层工具的边际收益不显著。

### 3.2 本轮新冒出的有效模式

**v1.x freeze 信号的提前识别**
- 自 v1.4.0 起 Plan agent 反复警告 surface 膨胀
- v1.7.0 Plan agent 给出明确节奏：v1.8 = 倒数第 2 minor / v1.9 = 终曲 / 并行起 v2.0 RFC
- v1.8.0 时正式接受这个节奏
- 实际节奏完全遵循 → v1.x 收尾秩序井然，没有「再加最后一个 feature」的诱惑

**4 个并发 Q 的最高峰**
- v1.7.0 bundled minor 一口气含 4 件（C1+C2+C3+C4），讨论阶段 4 个 Q-locks
- v1.8.0 也是 bundled
- 「Bundle vs Split 判断标准」（主题统一 + 无依赖 + < 1500 LoC）反复成立
- 但 v1.9.0 单一主题（BudgetExpand）保持纯净，没强行塞 patch — 终曲该干净

**closure-based refactor 在 freeze release 中的应用**
- v1.9.0 Commit 1 是 pure refactor：`wrapBudgetError` 改为 take partial-builder closure
- 不改任何 test 不改任何行为，只把"3 路径各自构造 PartialDiagnostics"的差异点抽出来
- 让 AskGlobal/AskDrift 增量极小（每路径只多一行 closure）
- 这是 v1.x 最后一次 add-without-mutate 的成功示范，给 v2.0 重构铺路

**跨仓 ROI 兑现：otel v0.3.0**
- v1.5.0 加 `OnGenerateUsage` hook 时 Plan agent 警告「foundation-only minor」
- 24 小时内 otel v0.3.0 真消费这个 hook → `rag.generate.tokens` OTel metric
- 用户开箱即得 per-stage token cost 视图
- 验证了「先加 hook，证明真有消费者后再让 hook 更复杂」的策略

**Plan agent 用 `--auto --merge` polling 接力**
- 第 7 轮下游同步首次出现 execute agent 中途返回（"background poller running"）
- 主线程接手 polling 直到 merge，再派 fresh agent 完成剩余
- 后续轮次的 brief 改为「**用同步 Bash polling loop，不要 run_in_background**」
- 跨 agent 接力流程清晰、零 commit 丢失

### 3.3 本轮的实际偏离与修正

**`scriptedAsker` 既有断言修订**
- v1.5.1 SubStages 改了 inner Generate 的 stage tag，导致 v1.5.0 已有断言「全部 `stage=="ask"`」**必须更新**
- execute agent 自报偏离：把断言收紧成「必须出现新 sub-stage tags」
- 判断合理：TDD 该这么做；brief 里"DO NOT modify v1.5.0 fixtures"被解读为 testdata/ 文件不动
- 修订后 brief 表述更明确：「现有测试若**因新功能本质上改变其断言对象**则可更新；fixture mock 实例**不可改**」

**`MaxTotalTokens` 6 站点 → 12+ 站点的接受度**
- v1.9.0 把 `wrapBudgetError` 从 6 站点扩到 12+
- Plan agent v1.8.0 时已警告「v2 反正要扔，accept duplication」
- v2.0 RFC Pillar 2 明确把这个改成 `AskState` machine
- **结论**：v1.x 终曲阶段接受「v2 会重做」的复制是诚实工程

**Plan agent Write 权限缺失（首次发现）**
- v2.0 RFC drafting 时 Plan agent 报告：「Plan 工具集中没有 Write 权限，无法落盘 RFC 文件」
- Plan agent 把 RFC 全文内联返回（~2900 词）
- 主线程用 Write 工具落盘
- **教训**：Plan agent 的工具集是 Read+Bash+Grep+Glob+WebSearch+WebFetch+mcp，**不含 Write**
- 适合做「设计返回文本」，不适合作「写文档到文件」的执行
- 下次类似任务直接派 general-purpose execute agent 或主线程亲自写

**baseline 测试数 off-by-N**
- 多次发生：brief 写 baseline N 但实际是 N±3~5
- v1.6.0：471 vs 471（准）
- v1.7.0：471→512 brief 估、实际 474→512（baseline off-3）
- v1.8.0：512 vs 509（off-3）
- v1.9.0：544 vs 549（off-5）
- execute agent 每次都 verify 真实值并诚实记录
- **结论**：人工估值不重要，CI gate 才是真，execute agent 自验即可

## 5. Lessons Learned（本段新增）

### 5.1 v1.x 出口节奏的工程

**「最后一个 minor」需要有明确叙事**
- v1.9.0 BudgetExpand 不是凑数 feature，而是**收 v1.7.0 显式 defer 的文档漏洞**
- CHANGELOG 明确标注 "Final additive v1.x minor"
- 引用 `docs/v2-rfc.md` 让 v1.9.0 → v2.0 路径在 git history 里就可解析
- 没有最后一刻塞 patch 的诱惑（C-Cache 已在 v1.8 落地，C-Drift 已在 v1.6）

**v2.0 RFC 应当与最后 minor 同步落稿**
- 早于 v1.9.0 落 → 引用关系还不存在
- 晚于 v1.9.0 落 → v1.9.0 CHANGELOG 引用的链接当时是 dangling
- **正好 v1.9.0 ship 后立即落 RFC** → CHANGELOG 链接立即可解析
- 本会话节奏：v1.9.0 PR #21 merge → RFC PR #22 merge（间隔 ~30 分钟）

**「3 PR 都进 master 再打 tag」节奏被否决**
- v1.7.0 Plan agent 推 3 PR 分批入 master 后打 tag
- discuss Q1 我反对：「部分 v1.7.0 commits 已暴露给下游但 untagged」是反模式
- 改回单 PR 单 tag。后续 v1.8 / v1.9 沿用单 PR
- **教训**：tag 之前 master 不该携带本 tag 范围内的未完成代码

### 5.2 技术层面（本段新增）

**Error 上挂 Partial-State 是 v1 范围内的最佳近似**
- v1.7.0 `BudgetExceededError.PartialDiagnostics` 把"半成品 Diagnostics"挂在 error struct
- 这是因为 `Ask` 返回 `(Answer, error)` 双值，没有「半成功」槽位
- v1.x additive 锁住了「改返回类型」的路
- v2.0 用 `AskResult.State` 状态机彻底重设——但 v1.7-v1.9 的 PartialDiagnostics 形态是 v1.x 范围里**唯一诚实的近似**
- **教训**：v1.x freeze 下，结构性问题用「最小可接受妥协」桥过去，把真改革留 v2

**Cache hit bypasses observer + budget 是设计取舍而非 bug**
- v1.8.0 Cache 命中时不 fire `OnGenerateUsage`、不消耗 `MaxTotalTokens` budget
- 这是 cache 本意：免费就是免费
- 但用户可能预期 cache hit 也有 cost telemetry
- CHANGELOG 和 godoc 双重明确这点（R5）
- **教训**：「这是设计选择」必须显式 documented；隐式可能被当 bug 报

**`normalize()` 跨包复制是 DRY 让位于零循环**
- v1.8.0 Cache 需要 normalize；`eval.normalize` 已存在但是 `eval` 包 unexported
- 选 (b) 在 `rag/grader_cache.go` 复制 1 行 normalize，**不开 `rag → eval` 反向 import 路径**
- 真的就 1 行 `strings.Join(strings.Fields(strings.ToLower(s)), " ")`，复制成本远低于 import 耦合
- **教训**：小函数的复制 < 耦合带来的 import cycle 风险

**Plan agent Write 权限缺失（架构限制）**
- v2.0 RFC drafting 暴露的 Plan agent 工具集**不含 Write/Edit**
- Plan agent 只能返回设计文本 + 由主线程或 execute agent 落盘
- **教训**：
  - 写代码：execute agent
  - 写设计/文档：主线程或 general-purpose execute agent
  - Plan agent 永远是「返回文本」，不直接写文件

### 5.3 协作层面（本段新增）

**「sharp questions」机制保持高 ROI**
- 本段 5 次 4-phase 都跑了 discuss 阶段，每次 4-5 个 Q-locks
- 用户全 ✅ 答复模式（"Q1✅ Q2✅..." 或 "A"）保持简洁
- **关键**：discuss 不是仪式，每次都至少抓住 1 个 Plan agent 没考虑到位的点
  - v1.6.0 Q3：NaN-touching = Undefined（不分 Improved/Regressed）
  - v1.7.0 Q3：partial 塞 error 不塞 Answer
  - v1.8.0 Q3：加便捷构造 NewCachingGrader
  - v1.9.0 间接：通过 v1.8 时 Plan agent 已警告 12-site 复制，v1.9 接受
- 这套机制即使 Plan agent 越来越熟练也不能省

**"a" vs "Q1✅..." 歧义事件**
- 一次 discuss 阶段用户答 "a" — 既可以是「全 ✅ 用我的倾向」也可以是「每题选 option(a)」
- 主线程明确问 clarify，没强行猜
- 用户答「A」（全 ✅）
- **教训**：单字符 reply 在多 Q 上下文有歧义时主动 clarify 优于赌

**跨 agent 接力的 polling 模式落地**
- 第 7 轮下游同步 execute agent 中途返回，主线程接手 polling 后再派新 agent
- 后续 brief 加入 polling pattern 模板（同步 Bash loop，不 background）
- 第 8/9 轮一次成功，无中断
- **教训**：execute agent 默认不擅长长时间等待；主线程或专门的 polling 步骤更稳
