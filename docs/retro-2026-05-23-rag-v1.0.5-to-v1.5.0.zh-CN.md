## 3. 流程评估

### 3.1 真正起作用的模式

**4-phase 工作流（分析→讨论→确定→执行）**
- 「讨论」阶段的 5 sharp questions 几乎每次都暴露至少 1 个 Plan agent 没考虑到位的细节（v1.3.0 的 NaN-JSON 不可序列化、v1.4.0 的 Parallelism=1 语义、v1.5.0 的 Jitter enum vs ratio 等）
- 主线程承担 orchestration + critique 角色，不亲自做设计——节省 context window，也让 critique 视角更独立
- 「确定」阶段强制把 Q1-Q5 决议落字面 lock，execute agent 严格不能 renegotiate

**design-then-execute pattern**
- 4 次 Plan agent 设计 + 5 次 execute agent 实施
- Plan agent 在 v1.3.0 抓到「eval.Asker 已存在」、v1.4.0 抓到「JudgeApplied 需独立标志」、v1.5.0 抓到「eval→rag.countingModel 是 import cycle」——这些都是 execute agent 蒙头干会踩的坑
- 单次 spawn = 单次孤立 context，但每次都喂完整 brief（包含此前所有锁定决议），保证一致性

**TDD 纪律**
- 每个 commit 都是 RED→GREEN 对，名字按行为命名（不是 unit-tested-by-coverage）
- v1.4.0 execute agent 出了一次「Commit 6 测试一写就 GREEN」（因为 Commit 5 已经包含了 abort 逻辑），诚实记录 + 测试保留为 regression gate——这是健康反应不是失败
- Race detector 全程 CLEAN，含 1000-goroutine 并发的 StageUsageAccumulator + Parallelism=4 的 benchmark

**真 merge commit + tag annotated**
- 9 次 PR 都是 `gh pr merge --auto --merge`，零 squash 零 rebase
- 9 个 tag 都是 annotated（不是 lightweight），body 带完整 CHANGELOG 段
- `git log --oneline v1.0.5..v1.5.0` 完整保留了 RED/GREEN 节奏，bisect 友好

**Bundle vs Split 节奏**
- v1.1.0(Track B) → v1.1.1 patch（GraderExample）→ v1.2.0：小补丁先行验证机制
- v1.2.0 → v1.2.1 patch（收 defer 项）→ v1.3.0：同模式
- v1.3.0 → v1.4.0 bundle（Judge+Par 紧耦合）：合理捆绑
- v1.4.0 → v1.5.0 bundle（CostObs+Retry 都是 production hardening）：合理捆绑
- 决策始终从「主题统一性 + 工作量 + 评审可控性」三维评估，没出过 PR 过大的事故

### 3.2 反复出现的小摩擦

**Intermediate apisnapshot RED 态**
- v1.2.0 execute agent 把 snapshot regen 留到 Commit 7，导致 C1-C6 中间状态 `internal/apisnapshot` 测试是红的
- v1.4.0 execute agent 自己识别这是问题，改为每个 commit 增量更新 snapshot——成为新基线模式
- v1.5.0 已经自然沿用增量模式
- 经验：**有 export 变更的 commit 必须同步更新 snapshot**，否则 bisect 跨该 commit 会撞红

**Round 索引 0-based vs 1-based**
- v1.1.1 GraderExample 设计时假设 0-based，execute agent 验证仓库实际 1-based（`ask.go:54` loop 从 1 起），调整后文档化在 doc comment
- 类似的 off-by-one 风险在 Plan→execute 衔接时反复出现，但 execute agent 每次都正确识别并报告
- 经验：**Plan agent 提供的字段语义假设，execute agent 必须验证实际代码再实施**

**Fixture 命名碰撞**
- v1.4.0 Commit 2 原定 `recordingJudge` 名字，execute agent 发现 `eval/global_test.go:40` 已用，改为 `recordingBenchmarkJudge`
- v1.5.0 类似检查发现 `recordingModel` 没碰撞，沿用
- 经验：**新 test fixture 命名前先 grep 同名**，零代价

**Plan agent 偶尔在 Out-of-Scope 重复 Locked Decisions**
- v1.4.0 Plan brief 里 Jitter 字段被描述为 ratio，execute agent 自己 reconsider 成 enum
- v1.5.0 Plan 已把 Jitter 显式锁为 enum，没再绕弯
- 经验：**Plan brief 模糊处 execute agent 会自己决定**，这种自决要标记 deviation 透明化（agent 都做了）

## 6. Lessons Learned

### 6.1 流程层面

**「Plan agent 设计 + execute agent 实施 + 主线程 critique」三角分工有效**
- Plan agent 不实施意味着它有空间真正考虑设计
- execute agent 不设计意味着它会一定程度遵循 brief，不擅改
- 主线程不做任一意味着可以站在用户视角 critique 双方

**4-phase 不可压缩**
- 「讨论」阶段 5 questions 看似流程开销，但**每次都至少捕捉 1 个 Plan agent 没考虑到位的细节**。8 次 4-phase 跑下来，估算节省的返工 >> 流程开销
- 「确定」阶段把 Q1-Q5 决议字面 lock，execute agent 不可绕过，避免 mid-execution drift

**Bundle vs Split 的判断标准**
- 主题统一 + 无依赖 + 总量 < 1500 LoC → bundle
- 主题混合 OR 有依赖 OR 总量 > 1500 LoC → split
- 历史几次决策回看都正确

### 6.2 技术层面

**v1.x additive-only 在小步快跑下能撑很久**
- 9 个 tag 0 removed lines，证明 additive 不是束缚而是创造性约束
- 但 Plan agent 反复警告 surface 体量——这是真问题，到 ~1500 行可能就该转 v2.0

**NaN sentinel + zero-value default 是 v1 additive 的高频武器**
- v1.3.0 BenchmarkMetrics、v1.4.0 Judge metrics、v1.5.0 StageTokenUsage 都用此模式
- 模式：新字段 zero 等价旧行为 + 不存在的统计语义用 NaN 而非 0 表达
- 让用户 demonstrably 知道「没启用 feature」vs「启用了但结果是 0」

**TDD RED-GREEN-immediate 是 healthy reality not failure**
- v1.4.0 Commit 6 写了 abort 测试一上来就 GREEN（因为 Commit 5 已经实现了对的语义）
- 测试保留作为 regression gate
- 比删除测试或硬把代码搞红更诚实

**race detector 是 invariant gate，不是事后检查**
- 5 处并发代码全程 race-clean
- 1000-goroutine accumulator 测试是 v1.5.0 的代价（写起来麻烦）但承诺了「并发用法是契约」
- 没有这个测试，v2.0 任何并发重构都会担心兼容性

### 6.3 协作层面

**explicit > implicit**
- v1.5.0 Q2「stage 经 ctx-value vs 显式参数」选了显式，理由就是 ctx-value 是 magic
- v1.4.0 Q3「JudgeMetrics flat vs sub-struct」选了 flat，理由是 sub-struct 只为 2 个字段过度抽象
- 反复选简单/显式那个，长期维护成本最低

**veto power matters**
- 每次「讨论」阶段我给 5 questions + 我的倾向，用户基本全 ✅ 但「我的倾向」是认真的——给了用户低成本 veto 任一项的能力
- 如果哪次 Plan agent 推的方向我自己也不确定，veto 倾向会标 ❓ 让用户主动选

**transparent deviations**
- execute agent 自报偏离（含 Round 索引、fixture 改名、Commit 5/6 合并等）让我能即时判断是否问题
- 没有 silent fix 也没有 silent break
